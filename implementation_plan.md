# ITSM(Service Now 형태) 시스템 구축 계획서

본 문서에서는 IT 서비스 관리(ITSM) 시스템 구축을 위한 기본 아키텍처 및 구현 계획을 제안합니다. 사용자의 피드백을 바탕으로 최신 트렌드에 맞는 Next.js 풀스택과 PostgreSQL을 활용하여, 고도화된 인시던트 상태 워크플로우(결재 포함) 및 상세 티켓 폼을 구현합니다.

## User Review Required

> [!IMPORTANT]
> 계획서를 더 상세하게 보실 수 있도록 개발 작업(코드 작성 등)은 진행하지 않고, 요청하신 **`/github/servicenow` 디렉토리 확인 및 Git 초기화**만 완료해 두었습니다. 
> 현재 계획서를 천천히 더 검토해 보시고, 추후 진행을 원하실 때 말씀해 주시면 개발(프레임워크 초기화 및 DB 연동)에 착수하겠습니다.

## Proposed Changes

### 기술 스택 및 환경 설정

- **작업 디렉토리:** `/github/servicenow` (GitHub `ServiceNow` 리포지토리 연동)
- **Frontend & Backend (Full-stack):** Next.js (React) + TypeScript
- **Styling:** Vanilla CSS (모던 UI/UX 적용)
- **Database:** PostgreSQL 
- **인증 (Authentication):** 권한(Role) 기반 로그인 (관리자 / 팀장 / 장애 처리자 / 일반 사용자)
- **파일 스토리지:** 티켓 스크린샷 및 첨부파일 저장을 위한 로컬/Cloud Storage 연동 구조 준비

### 단계별 개발 계획

#### Phase 1: 환경 구성 및 기반 시스템 (Foundation)
- **인프라 및 리포지토리 셋업:** `sudo` 권한(`!wjdgns2`)을 사용하여 `/github/servicenow` 폴더 생성 및 Git 로컬 리포지토리 초기화 완료 (향후 GitHub 연동 진행). Node.js, npm, PostgreSQL 설치 예정.
- **DB 스키마 구성:** 회사, 모듈(FI, CO, SD, MM, PP, BC 등), 사용자/팀장 정보, 티켓 본문 및 첨부파일 메타데이터 테이블 구성
- **기본 레이아웃:** ITSM 전용 대시보드 구축 및 공통 디자인 시스템 적용

#### Phase 2: 인시던트 티켓 등록 및 고도화된 워크플로우 (우선 구현 핵심 기능)
- **티켓 폼(Form) 고도화:**
  - 기본 정보: 회사, 모듈(FI, CO 등), 제목, 내용
  - 추가 정보: 우선순위(`보통`, `긴급`, `매우긴급`), 완료 예정일 입력란
  - 미디어: 스크린샷 이미지 첨부 및 다중 첨부파일 등록 기능
- **상태 처리 및 승인 워크플로우 (엄격한 상태 전이 로직):**
  - **일반 사용자(User):** `임시저장` / `등록` / `반려` / `완료` 상태 변경 권한
    - 최초 작성 시 `임시저장` 또는 `등록` 처리.
    - 처리자가 `고객확인` 상태로 넘긴 티켓에 대해 검토 후 최종 `완료` 또는 `반려` 처리.
  - **결재 로직 (팀장 승인):**
    - 사용자가 티켓을 `등록` 상태로 변경하면 팀장에게 승인 요청됨.
    - 팀장이 승인할 경우 `접수대기` 상태로 자동 변경.
    - *옵션 로직:* 사용자의 팀장이 지정되어 있지 않은 경우, `등록` 즉시 자동으로 `접수대기` 상태로 변경됨.
  - **장애 처리자(Agent):** 
    - `접수대기` 상태인 티켓만 조회 및 접수 가능 (`임시저장`이나 `등록` 상태의 티켓은 접수 불가).
    - `접수대기` ➔ `접수완료` ➔ `해결중` ➔ `고객확인` 순으로만 상태 변경 가능.
  - **관리자(Admin):** 모든 티켓의 상태, 내용, 권한을 강제로 제어할 수 있는 슈퍼유저 권한 보유.

#### Phase 3: 집계 및 통계 대시보드 (Expansion)
- **월간 통계 및 그래프 시각화:** 최근 한 달간의 티켓 처리 현황을 한눈에 볼 수 있는 대시보드
- **다차원 집계표 제공:**
  - 회사별 발생/처리 현황
  - SAP/시스템 모듈별(FI, CO, SD, MM, PP, BC 등) 현황
  - 장애 처리자(Agent)별 처리 건수 및 상태(완료율 등) 집계표 및 차트 제공

## Verification Plan

### Automated Tests
- 제공받은 비밀번호를 통한 PostgreSQL 및 Node.js 무인(Non-interactive) 설치 검증
- 첨부파일 업로드 API 및 티켓 상태 전이(State Machine)에 대한 백엔드 로직 테스트 (팀장 유무에 따른 상태 스킵 로직 검증 포함)

### Manual Verification
- **워크플로우 테스트 시나리오:**
  1. 일반 유저로 로그인하여 첨부파일, 우선순위를 포함한 티켓을 `등록` 상태로 생성.
  2. 팀장 계정으로 접속하여 해당 티켓을 승인해 `접수대기`로 변경되는지 확인.
  3. 처리자(Agent) 계정으로 접속하여 해당 티켓을 `접수완료` -> `해결중` -> `고객확인`으로 변경.
  4. 다시 일반 유저로 접속하여 티켓을 `완료` 처리하는 End-to-End 테스트 진행.
- 대시보드의 회사/모듈별 통계 그래프 시각적 검증
