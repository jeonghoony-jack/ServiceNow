# ITSM(Service Now 형태) 시스템 구축 계획서

본 문서에서는 IT 서비스 관리(ITSM) 시스템 구축을 위한 기본 아키텍처 및 구현 계획을 제안합니다. 사용자의 피드백을 바탕으로 최신 트렌드에 맞는 Next.js 풀스택과 PostgreSQL을 활용하여, 고도화된 인시던트 상태 워크플로우(결재 포함) 및 상세 티켓 폼을 구현합니다.

## User Review Required

> [!IMPORTANT]
> 계획서를 더 상세하게 보실 수 있도록 개발 작업(코드 작성 등)은 진행하지 않고, 요청하신 **`/github/servicenow` 디렉토리 확인 및 Git 초기화**만 완료해 두었습니다. 
> 현재 계획서를 천천히 더 검토해 보시고, 추후 진행을 원하실 때 말씀해 주시면 개발(프레임워크 초기화 및 DB 연동)에 착수하겠습니다.

## Proposed Changes

### 기술 스택 및 환경 설정

- **작업 디렉토리 구조:** 
  - `/github/servicenow/` : ITSM 기획/설계 문서 및 Git 관리
  - `/github/next-js/` : 시스템 전역 통합 Next.js 풀스택 환경
- **Frontend & Backend (Full-stack):** Next.js (React) + TypeScript (모바일 웹 환경에서도 원활하게 접속 가능하도록 반응형(Responsive) 웹 디자인 적용)
- **Styling:** Vanilla CSS (모던 UI/UX 적용)
- **Database:** PostgreSQL 
- **인증 (Authentication):** 초기에는 DB 기반 인증(권한/Role 기반 로그인)으로 구축하며, 향후 SAML2 SSO 인증이 가능하도록 확장성을 고려하여 설계합니다. (관리자 / 팀장 / 장애 처리자 / 일반 사용자)
- **파일 스토리지:** 티켓 스크린샷 및 첨부파일 저장을 위한 로컬/Cloud Storage 연동 구조 준비 (용량 제한 및 보존 기한(Retention) 정책 고려)

### 단계별 개발 계획

#### Phase 1: 환경 구성 및 기반 시스템 (Foundation) - **기본 인프라 완료**
- **인프라 및 공통 웹 환경 셋업 (완료):**
  - `/github/servicenow` 폴더 및 Git 저장소 초기화 완료.
  - 시스템 전역에 활용될 통합 Next.js 풀스택 환경을 `/github/next-js` 경로에 성공적으로 구축.
  - 공통 Database인 PostgreSQL 16 환경 구성 완료.
- **DB 스키마 구성:** 회사, 모듈 및 처리 구분(관리자 페이지에서 동적 관리), 사용자/팀장/처리자 정보(연락처 포함), 티켓 본문 및 첨부파일 메타데이터 테이블 구성
- **기본 레이아웃:** ITSM 전용 대시보드 구축 및 공통 디자인 시스템 적용

#### Phase 2: 사용자 관리 및 고도화된 인시던트 워크플로우 (우선 구현 핵심 기능)
- **관리자 전용 사용자/모듈 등 기준정보 관리 시스템:**
  - **사용자 관리 페이지:** 일반 사용자, 결재자(팀장), 장애 처리자 등 통합 관리. 모든 사용자에 대해 메일, 전화번호 등 기본 연락처 정보 입력 필수.
  - **고객사별 관리:** 사용자와 결재자는 특정 고객사(Company)별로 구분 및 관리.
  - **모듈 및 처리 구분 동적 관리:** 관리자 페이지에서 '시스템 모듈(FI, CO 등)' 뿐만 아니라 '처리 구분(프로세스 개선, 신규 개발, 권한 등)' 데이터도 별도 테이블로 구성하여 동적으로 추가/수정/삭제 가능.
  - **모듈 담당자 설정:** 각 장애 처리자(Agent)가 담당하는 모듈을 설정 가능.
- **티켓 폼(Form) 고도화:**
  - **단일 통합 폼:** 사용자, 팀장(결재자), 처리자(Agent) 모두 동일한 필드로 구성된 통일된 화면 폼을 사용.
  - **기본 정보:** 제목, 고객사(일반 사용자의 경우 본인 소속 회사 자동 지정), 요청자, 연락처, 이메일
  - **분류 정보 (드롭다운):** 모듈(FI, CO 등), 처리 구분(프로세스 개선, 등). 사용자와 처리자는 필요에 따라 드롭다운으로 값을 선택 및 변경 가능.
  - **상세 정보:** 우선순위(`보통`, 긴급`, `매우긴급`), 완료 희망일
  - **내용 및 첨부:** 요청 내용(본문), 다중 파일 첨부 및 스크린샷 이미지 첨부 기능
- **상태 처리 및 승인 워크플로우 (엄격한 상태 전이 로직):**
  - **일반 사용자(User):** `임시저장` / `등록` / `반려` / `완료` 상태 변경 권한
    - 최초 작성 시 `임시저장` 또는 `등록` 처리.
    - 처리자가 `고객확인` 상태로 넘긴 티켓에 대해 검토 후 최종 `완료` 또는 `반려` 처리.
  - **결재 로직 (팀장 승인):**
    - 사용자가 티켓을 `등록` 상태로 변경하면 팀장에게 승인 요청됨.
    - 팀장이 승인할 경우 `접수대기` 상태로 자동 변경.
    - *옵션 로직:* 사용자의 팀장이 지정되어 있지 않은 경우, `등록` 즉시 자동으로 `접수대기` 상태로 변경됨.
  - **장애 처리자(Agent):** 
    - `접수대기` 상태인 티켓만 조회 및 접수 가능 (`임시저장`이나 `등록` 상태의 티켓은 접수 불가).
    - `접수대기` ➔ `접수완료` ➔ `해결중` ➔ `고객확인` 순으로만 상태 변경 가능.
    - **답변 관리 기능:** 담당 티켓에 대해 답변을 [임시저장 / 삭제 / 저장 / 수정] 할 수 있는 독립적인 관리 기능 제공.
  - **관리자(Admin):** 기준정보(사용자/고객사/모듈/처리구분 등) 관리 및 모든 티켓의 상태, 내용, 권한을 강제로 제어할 수 있는 슈퍼유저 권한 보유.
- **알림 및 이메일 발송 기능:**
  - 티켓 상태 전이(예: 승인 요청, 고객 확인 등) 발생 시 담당자에게 이메일 및 시스템 알림(Notification) 발송

#### Phase 3: 집계 및 통계 대시보드 (Expansion)
- **월간 통계 및 그래프 시각화:** 최근 한 달간의 티켓 처리 현황을 한눈에 볼 수 있는 대시보드
- **보고서 추출 기능:** 집계된 데이터를 바탕으로 월간 보고서 및 연간 보고서용 자료 추출(Export) 기능 제공
- **다차원 집계표 제공:**
  - 회사별 발생/처리 현황
  - SAP/시스템 모듈별 현황
  - **처리 구분별** 및 **우선순위별** 현황
  - 장애 처리자(Agent)별 처리 건수 및 상태(완료율 등) 집계표 및 차트 제공

## Verification Plan

### Automated Tests
- 제공받은 비밀번호를 통한 PostgreSQL 및 Node.js 무인(Non-interactive) 설치 검증
- 고객사별 사용자 권한 분리 및 첨부파일 업로드 API, 티켓 상태 전이(State Machine)에 대한 백엔드 로직 테스트 (팀장 유무에 따른 상태 스킵 로직 검증 포함)

### Manual Verification
- **워크플로우 및 권한 테스트 시나리오:**
  1. 관리자 계정으로 모듈 및 처리구분 추가, 그리고 각 회사별 일반 유저, 팀장, 장애 처리자 생성 (연락처 및 담당 모듈 지정).
  2. 일반 유저로 모바일 웹 화면에서 로그인하여 단일 통합 폼 양식을 통해 첨부파일, 우선순위, 처리구분 등을 포함한 티켓을 `등록` 상태로 생성.
  3. 팀장 계정으로 접속하여 해당 티켓을 승인해 `접수대기`로 변경되는지 확인.
  4. 해당 모듈의 처리자(Agent) 계정으로 접속하여 해당 티켓을 `접수완료` -> `해결중` -> `고객확인`으로 변경 및 답변 임시저장/수정, 필요시 모듈/처리구분 드롭다운 변경 동작 검증.
  5. 다시 일반 유저로 접속하여 티켓을 `완료` 처리하는 End-to-End 테스트 진행.
- 대시보드의 다차원 집계표(모듈별, 회사별, 처리구분별, 우선순위별) 시각적 검증 및 월간/연간 보고서 추출 확인
